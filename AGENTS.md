# Repository Guidelines

## Project Structure & Module Organization

This is a Python call-center analytics and reporting project. Core pipeline code lives at the repository root:

- `generate_report.py`: main report entry point.
- `call_log_analyzer.py`: cleaning, metrics, chart generation, and database persistence.
- `cleaning.py`, `validate_historical.py`, `weekly_data_manager.py`, `store_snapshot.py`: supporting pipeline modules.
- `templates/`: Jinja2 HTML report templates.
- `data/`: raw and combined CSV inputs, named like `CallLogLastWeek_*.csv` and `AbandonedCalls*.csv`.
- `reports/`: generated HTML, CSV, and verification outputs.
- `sanity/`: executable validation scripts and audit helpers.
- `reference/`, `sample/`, and `archive/`: comparison material, sample data, and older/debug scripts.

## Build, Test, and Development Commands

Install dependencies:

```bash
pip install -r requirements.txt
```

Generate the main report:

```bash
python generate_report.py
```

Run focused validation checks:

```bash
python sanity/verify_data_integrity.py
python sanity/test_core_logic.py
python sanity/verify_metrics.py
```

Check Python syntax for edited modules:

```bash
python -m py_compile call_log_analyzer.py generate_report.py
```

## Coding Style & Naming Conventions

Use 4-space indentation and standard Python naming: `snake_case` for functions, variables, and modules; `UPPER_CASE` for constants. Keep data-frame column handling explicit and readable because report totals depend on exact field names. Prefer small helper functions for shared date, week, and classification logic. Do not commit generated bytecode (`__pycache__/`, `*.pyc`) or local secrets.

## Testing Guidelines

There is no formal pytest suite yet; use the scripts in `sanity/` as regression checks. Run the relevant checks after changing week assignment, customer classification, abandoned-call logic, report totals, or database persistence. When adding new checks, place them in `sanity/` and name them by intent, for example `verify_trade_abandonment.py` or `test_core_logic.py`.

## Commit & Pull Request Guidelines

Recent history uses short, imperative commit messages such as `update latest files` and `latest updates`. Prefer more specific summaries when possible, for example `fix call ID upsert migration` or `refresh weekly report data`. Pull requests should describe the changed pipeline behavior, list validation commands run, and call out generated data/report files separately from source changes.

## Security & Configuration Tips

Keep credentials in `.env` and document placeholders in `.env.example`. Database code loads settings through `python-dotenv`; avoid hard-coding passwords or host-specific overrides. Treat `reports/` and `data/` as potentially sensitive because they can contain call details and customer phone numbers.
